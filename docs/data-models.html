This project uses RedisJson to do CRUD operations on its data through the redis-om-python library.


There is a one to many relationship between users and contacts; a user can have multiple contacts. In NOSQL, we can create two collections, namely User and Contact. The Contact collection contains enough information to populate the contact list, a list of user.


Additionally, the User collection contains more information used to populate the user profile details view.


Note: You can improve this approach by embedding User details within the contact list. However, this pattern didn't work in the current version of the redis-om-python library.

Likewise, There is a many to many relationship between users and messages; a user can have multiple conversation with other users. Each conversation contains enough information to populate the chat view. In NOSQL, we can create three collections, namely User, Conversation, and Message. The Conversation collection contains enough information to populate the messages list between each sender and receiver.


Note: Similarly, we can improve this approach by embedding messages' details within the messages list of the conversation. However, this pattern didn't work in the current version of the redis-om-python library.

There is a one to many relationship between users and rooms; a user can exist in one or more rooms. There is also one to many relationship between rooms and conversations; a room can have one or multiple conversations. Each room contains enough information to populate the chat view. here, we can create three collections, namely Room, Conversation, and Message. The Room collection contains enough information to populate the messages list between a sender and multiple receivers. The only difference here from the previous users-messages model is that the receiver side of the conversation is the room itself.
